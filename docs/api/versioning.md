# API Contract Versioning

This guide defines how to evolve and maintain the contract documentation under `docs/api`.

## Goal

Contract docs should allow humans and AI agents to answer these questions reliably:

- what can be sent to this endpoint or workflow?
- what is guaranteed in the response or emitted payload?
- what fields are required, optional, or conditional?
- what changed, and is the change backward compatible?

## Source of Truth Rules

- the implementation and `docs/api` must match exactly
- any API contract, workflow contract, webhook payload, request model, response model, or integration payload change must update the relevant `docs/api` files in the same task
- if a contract change affects FastAPI endpoints, update both the markdown docs and in-code OpenAPI metadata/model descriptions when appropriate

## What Counts as a Contract Change

A documentation update is required when changing:

- endpoint request fields
- endpoint response fields
- status codes or error semantics
- webhook payload fields
- workflow inputs or outputs
- integration request/response payloads
- required environment/auth prerequisites
- deterministic file naming or object-path rules relied upon by consumers

## Compatibility Categories

## Backward Compatible

Examples:

- adding a new optional response field
- clarifying existing docs without changing behavior
- adding a new endpoint without changing existing ones
- adding a new optional request field that existing clients may ignore safely

## Potentially Breaking

Examples:

- renaming a field
- removing a field
- changing a field type
- changing default behavior in a way clients observe
- changing file naming or storage path conventions
- changing webhook envelope structure
- changing error status codes relied upon by clients

## Documentation Conventions for Changes

When updating a contract doc:

- keep examples concrete and realistic
- explicitly mark fields as required, optional, or conditional when that is not obvious
- document prerequisites and invariants near the contract, not only in prose elsewhere
- update related docs when a shared contract is referenced from multiple places

## Required Cross-Checks Before Finishing

For each contract-related change, verify:

- endpoint docs still match router behavior
- model reference docs still match Pydantic models
- workflow docs still match Temporal workflow/activity behavior
- integration docs still match service clients and emitted payloads
- `docs/api/README.md` still indexes the current document set

## Change Logging Convention

This repository does not currently maintain a formal API version number in code.

Until that changes, use this documentation convention:

- describe the new behavior directly in the affected doc
- if the change is materially breaking, add a short `## Breaking Change Note` section in the affected document
- update all related examples to the new canonical form in the same task

### Example Breaking Change Note

```markdown
## Breaking Change Note

As of this change, `workflow_id` is generated by the server for this endpoint and any client-supplied value is ignored.
```

## Recommended Future Enhancement

If contract churn increases, add a repository-level change log such as:

```text
docs/api/CHANGELOG.md
```

Suggested entries:

- date
- affected docs
- compatibility classification
- short migration note

## Review Checklist

Before considering a contract task complete:

- [ ] implementation matches docs
- [ ] docs match examples
- [ ] related shared-model docs are updated
- [ ] related integration/workflow docs are updated
- [ ] index references are updated
